# 交叉审计矩阵

> 本文件由 CLAUDE.md [总体规则] 审计关闸引用。
> 定义 5 层 × 5 维审计矩阵的完整执行规范。

---

## 触发时机

- Product-Spec.md 定稿后 → L1 审计（需求层自洽）
- Design-Brief.md 定稿后 → L1-L2 审计（需求↔设计交叉）。如 Design-Brief 被跳过 → 所有含 L2 的审计单元自动标记 SKIP
- 设计稿完成 .pen 定稿后 → L1-L3 审计（需求↔设计↔元规则交叉）。如设计稿被跳过 → 所有含 L3 的审计单元自动标记 SKIP
- DEV-PLAN.md 生成后 → L1-L4 审计（全链路交叉）
- 每个 Phase 代码完成后 → L1-L5 审计（含代码层）
- 项目关联的 Skill 包（题材包/功能包）变更后 → L4 专项审计
- 用户提出"复查""检查一致性""有没有问题"等指令时 → 对应层级全面审计

**定稿定义**：Agent 完成该阶段全部产出 + 用户明确肯定（"可以了""没问题""继续"等）。未经用户确认 ≠ 定稿 → 不触发审计。

**层缺失处理**：L2/L3 为可选层。层不存在时，审计范围中涉及该层的所有 L×D 单元自动标记 `SKIP: 层未创建`，不计入 🔴/🟠/🟡 统计。跳过前告知用户后果（无设计参照 → 视觉一致性无法保证）。

## 5 层审计对象

| 层级 | 名称 | 涵盖文件 |
|------|------|---------|
| L1 | 需求层 | Product-Spec.md + Product-Spec-CHANGELOG.md（如 CHANGELOG 尚未创建——首次 Spec 定稿时——L1 审计中涉及 CHANGELOG 的检查项自动跳过，CHANGELOG 在审计第六步由 Agent 创建） |
| L2 | 设计层 | Design-Brief.md + .pen 设计稿文件 |
| L3 | 元规则层 | CLAUDE.md + agents/*.md（implementer/code-reviewer/feedback-observer/evolution-runner）+ EVOLUTION.md + feedback/ 目录 + workflows.md |
| L4 | Skill 层 | skills/*/SKILL.md（全部 Skill）+ 项目关联的 Skill 包（CLAUDE.md + SKILL.md + 模板 + 示例） |
| L5 | 代码层 | 项目源代码（适用时，含 package.json + src/ + 配置文件） |

---

## 5 维审计维度

- **D1 可追溯性**：上层需求在下层是否有对应实现或覆盖，逐功能点打勾打叉
- **D2 内部一致性**：同层文件之间的数值、术语、行为描述是否矛盾（术语表+数值表对照）
- **D3 完整性**：是否覆盖所有状态（空态/加载态/错误态/激活态/边界）× 所有异常路径 × 所有交互矩阵
- **D4 引用有效性**：所有文档中出现的文件路径、Skill名、Agent名、函数/组件名是否在磁盘/注册表中真实存在
- **D5 交互安全性**：多写者冲突策略、多入口行为差异约定、跨引擎数据桥接方案是否已明确定义

## 审计矩阵（层×维必须检查）

```
            D1追溯  D2一致  D3完整  D4引用  D5交互
L1 需求层     —       ✓       ✓       ✓       ✓
L2 设计层     ✓       ✓       ✓       ✓       ✓
L3 元规则层   ✓       ✓       ✓       ✓       ✓
L4 Skill层    ✓       ✓       ✓       ✓       ✓
L5 代码层     ✓       ✓       ✓       ✓       ✓
```
✓ = 必须检查；— = 该层为最上层，无可追溯对象

## 审计执行流程

### 第一步：确定审计范围
根据当前项目阶段，列出所有适用的 L×D 审计单元。声明跳过哪些单元及理由（如 L5 代码层尚未创建）。

### 第二步：逐单元检查
按 L1→L4 从上到下、D1→D5 从左到右逐格审计。
每个发现标注：严重度（🔴阻断 🟠高 🟡中 🟢低）+ 涉及文件 + 行号/位置 + 具体证据。

### 第三步：输出审计报告
按严重度分组，每项附修复建议。
统计各严重度数量、各层问题数、各维问题数（热力图）。

### 第四步：用户确认修复范围
阻断级和高风险必须修复。中风险逐条标注处理策略（修复/延期/接受）。
等待用户确认后执行修复。

### 第五步：执行修复
按严重度从高到低逐条修复。涉及多文件时同步修改保持一致性。
修复完成后对受影响维度执行增量审计。

### 第六步：放行
所有 🔴🟠 已修复、🟡 有明确处理策略 → 审计通过。
审计报告追加到 Product-Spec-CHANGELOG.md。

## 各维度关键检查项

### D1 可追溯性
- Spec 每个功能点 → Design Brief 对应描述（缺=未覆盖）
- Spec 每个功能点 → DEV-PLAN Phase/Task（缺=漏排）
- Design Brief 每条视觉规范 → 设计稿对应节点（缺=漏设计）
- DEV-PLAN 每个 Task → 代码文件（缺=漏开发）
- 设计稿每个按钮/组件 → Spec 功能定义（反向追溯，缺=多余设计）
- **DEV-PLAN 每个含 UI 的 Phase → 设计帧（反向追溯，缺=Agent 凭空创造界面，🔴阻断）**
- **代码中每个 UI 交互元素 → 设计帧/Spec（反向视觉检测，代码有但设计无=多余元素，audit-runner 自动扫描）**
- Agent 指令中的每个行为描述 → Spec 中对应的功能点

**L1 需求层语义自洽**（Agent 手工确认——Spec 内部逻辑无矛盾）：
- **章节/编号连续性**：Spec 中功能章节编号是否连续无断裂（如 §1 → §3 缺 §2）？发现断裂 → 🟠
- **总述与例外一致性**：某处说"所有 X 都做 Y"，后文说"Z 情况下除外"——总述处是否已标注例外引用？未标注 → 🟡
- **交互上下文明确性**：每个交互描述是否明确了发生的页面/视图上下文（如"在 Skill 主页中"vs"在工作台中"）？上下文不明确 → 🟠
- **元素归属正确性**：每个 UI 元素（右键菜单、弹窗、按钮）是否挂在正确的功能域下（如"项目管理"不是"技能包管理"的子项）？归属错误 → 🔴
- **矩阵与专项描述一致**：状态转移矩阵 / entry×行为矩阵 是否覆盖了所有专项描述的交互路径？矩阵有遗漏 → 🔴
- **操作按钮的重复/冲突检查**：点击 A → 打开标签页；若标签页已存在 → 是切换还是新建重复标签页？未定义 → 🟠
- **窗口/页面关闭行为**：每个独立页面的关闭按钮行为是否已定义（退出应用？返回上级？隐藏到托盘？）？未定义 → 🟠

### D3 交互行为完整性（Agent 确认清单）
- **展开/收缩操作**：展开后保存/取消不应自动关闭面板，需显式操作才改变状态
- **异步操作反馈**：所有网络/AI请求有加载态、成功态、失败态
- **破坏性操作**：删除/覆盖/清空有二次确认弹窗
- **拖拽操作**：拖拽完成后顺序持久化，刷新不丢失
- **输入操作**：所有输入框有 placeholder/默认值，空输入提交有校验
- 注：此清单由 `audit-runner` 自动提取候选，Agent 逐条确认判定

### D3 导航完整性
- 每个独立页面（非主页视图）必须有返回主页/工作台的路径
- 弹窗和对话框必须有明确的关闭路径（取消/关闭按钮或点击遮罩）
- 多步骤流程（如 Skill 导入向导）必须有"上一步/下一步"导航
- 注：`audit-runner` 自动检测 Spec 中定义了但缺返回路径的页面

### D2 内部一致性

**L1 需求层内部自洽**（Spec 内部数值/术语/行为无矛盾）：
- **状态定义一致性**：同一状态（⏹/⟳/✅）在所有描述处的判定条件是否一致？不一致 → 🔴
- **数值引用一致性**：同一数值（面板宽度、分隔线粗细、字号）在 Spec 不同位置是否相同？不同 → 🟠
- **术语统一**：同一概念在 Spec 内部用词是否一致（阶段卡片/阶段流程卡片、项目/Skill项目）？不一致 → 🟡
- **异常路径覆盖**：每个"自动/默认"行为是否定义了失败时的回退路径？（"自动保存"→保存失败？"自动加载"→加载失败？）未定义 → 🟡
- **全局交互模式一致性**：空状态/错误状态/加载态的 UI 模板是否统一定义（而非每个功能各自描述）？未统一 → 🟡

**跨文档一致**（原有）：
- 术语统一：同一概念在所有文档中的用词是否一致（阶段/Phase、文件映射/绑定、Skill/技能包）
- 数值统一：同一参数跨文档对比（面板宽度、保存间隔、动画时长、端口号）
- 格式统一：所有 SKILL.md frontmatter 是否符合相同 schema（name/version/description）
- 行为统一：同一交互在不同文档中的描述是否矛盾（固定 vs 可拖拽、自动 vs 手动）
- Agent/Skill 对齐：Agent 的指令范围是否与其绑定 Skill 的能力范围一致
- Skill 版本一致：所有 SKILL.md 的 version 字段是否符合语义化版本规范
- **L5↔L2 视觉一致性**（UI Phase 必须执行，不可跳过）：
  - 代码产出的每个页面/组件，是否与 .pen 设计稿中对应帧的视觉效果一致？
  - 检查项：布局结构（flex方向/nesting层级）、尺寸（width/height/padding/gap）、颜色（fill/stroke/opacity）、字体（fontFamily/fontSize/fontWeight）、圆角（cornerRadius）、边框（strokeWidth/strokeAlignment）
  - 检查方法：逐组件对照设计稿帧的精确数值→读取代码中的 Tailwind class 或 style→比对偏差
  - 不要求像素级逐个 px 的精确匹配，但布局结构、关键尺寸（面板宽度、工具栏高度）、颜色语义（$accent/$bg-panel）必须一致
  - 设计稿中定义的状态变体（hover/active/disabled/empty/loading/error）是否全部在代码中实现？
  - 设计稿中定义的交互行为（点击跳转、弹窗触发、面板切换）是否在代码中正确连线？

### D3 完整性

**L2 设计帧结构对应**（设计稿→代码 区块级覆盖——防"数值对但结构错"）：
- 每个设计帧的命名子区块（如"回收站面板""页面标题区"）在对应 Phase 代码中是否有实现？
- Phase 感知：仅检查当前 Phase 已交付的帧。未来 Phase 的帧自动跳过。
- 缺失子区块 → 🔴。此检查由 `l2-pen-check.js` 自动执行，数据来自 `.pen-frames.json` 缓存（通过 MCP 读取 .pen 生成）。
- 注：缓存需包含 `structure.allElements` 字段（子节点名称列表）。换产品后重新通过 MCP 生成缓存即可。

**L1 需求层完整**（Spec 无遗漏维度——对标行业产品规范）：
- **反功能需求**（专业版强制）：是否包含"不做"列表？每项是否有评估理由？遗漏 → 🟡
- **性能预算表**（专业版强制）：核心操作是否定义了 P50/P90/P99 目标？CI 是否在 P90 超预算时失败？缺失 → 🔴
- **降级路径时序图**（专业版）：关键并发路径（用户编辑↔自动保存↔AI 写入）在时间轴上的交错点是否可视化？缺失 → 🟠
- **恢复策略参数**（专业版）：API 失败退避是否定义了具体间隔/次数/每次 UI 态变化？仅写"重试"无参数 → 🟡
- **异步操作时序**：每个异步操作（重命名、删除、AI 生成、文件读写的 IPC 调用）是否定义了加载态→成功态→失败态的完整时序？缺失任一条 → 🟠
- **平台约束兼容**：是否明确标注了 Electron 特有的约束（如 `prompt()`/`confirm()` 不可用、`safeStorage` 必须用于 Key 存储）？未标注 → 🟡
- **窗口生命周期**：是否定义了窗口的创建/最小化/关闭/退出的完整行为？未覆盖 → 🟠

**结构层**（骨架——有没有定义）：
- 状态机完整：每个状态的 enter/exit 条件 + 触发事件 + 默认状态
- 空状态覆盖：所有展示区域在无数据时的表现
- 加载状态覆盖：AI 生成中、文件读取中、模型列表拉取中、章节索引解析中、OAuth 登录中
- 错误状态覆盖：API 连接失败、文件读写失败、模型不可用、Skill 导入失败、备份恢复失败
- 异常路径覆盖：用户做非预期操作时行为（重复点击、中断生成、关闭未保存、外部修改文件）
- 权限边界：AI 可访问/不可访问的文件范围、沙箱边界
- **动态内容溢出防护**：识别所有承载动态内容的区域，验证溢出处理机制是否存在
  - 文本编辑区（章节内容可扩写到 3000+ 字）→ 必须有滚动条或 auto-scroll
  - 列表类区域（Skill 列表、项目列表、提供商列表、章节列表、模型列表）→ 项数不确定时必须有滚动条
  - 标签栏（用户可打开 10+ 标签）→ 必须有横向溢出处理（滚动箭头/折叠菜单）
  - AI 对话区（单条回复可达数千字）→ 必须有 `overflow-y: auto`
  - 文件树（项目文件数不确定）→ 必须有滚动条
  - 预览模式（扩写后正文高度不可预测）→ 必须有滚动条
  - 回收站（可累积大量删除项）→ 必须有滚动条或分页
  - 判定标准：凡内容来源于用户输入或 AI 生成的区域，一律视为动态内容，必须有溢出保护
  - **主题可访问性**：面向长时间阅读/写作场景的产品（阅读器、笔记工具、写作工具、代码编辑器等），必须覆盖深色 + 浅色双主题
    - 判定标准：设计稿需包含至少一帧浅色模式关键页面，或 Design-Brief 中明确定义浅色反转策略（基于 CSS 变量自动生成，非独立设计）
    - Design-Brief 必须注明设计变量是否已覆盖浅色模式的颜色映射
    - 纯管理后台/配置面板类产品可豁免（如 API Key 管理页），但面向终端用户的主交互界面不可豁免
  - **量产工程基础**（面向终端用户的桌面/Web 产品，v1 可标注延期但不可遗漏）：
    - **无障碍 (a11y)**：键盘导航是否覆盖全部交互？色彩对比度是否满足 WCAG AA（正文 4.5:1，大号文字 3:1）？屏幕阅读器是否能正确朗读页面结构？
    - **国际化 (i18n)**：UI 字符串是否集中管理（非硬编码）？目标用户为非中文群体时是否已规划多语言方案？v1 中文 only 可标注豁免
    - **隐私合规**：用户数据收集范围是否明确？是否有隐私政策（至少模板）？是否提供用户数据导出/删除路径？Electron 应用是否使用 safeStorage 加密敏感数据？
    - **性能基线**：核心交互是否定义了性能目标（编辑器加载 < 100ms、AI 流式渲染 < 50ms）？是否有自动化性能回归测试？
    - **错误监控**：是否接入了崩溃上报（Electron crashReporter 或 Sentry）？前端错误是否有全局捕获和上报？
    - **自动更新**：Desktop 应用是否配置了 electron-updater + 更新服务器（GitHub Releases 至少）？用户是否知道如何获取新版本？
  - **[专业版] 设计系统深度**（对标 Linear/Obsidian 体验水准）：
    - **动效语义表**：是否按用途（入场/离场/注意/状态过渡）分类？是否有编排规则（stagger 间隔、父→子延迟）？缺失 → 🟠
    - **prefers-reduced-motion**：所有动效是否定义了即时切换替代方案？遗漏 → 🔴（a11y 底线）
    - **对比度内建校验**：设计 tokens 是否自带 WCAG 对比度计算（正文 AA 4.5:1，大号文字 AA 3:1）？边界不达标的 token 是否已标记？缺失 → 🟠
    - **focus-ring 样式**：设计 tokens 是否定义了 `--focus-ring-color/width/offset`？缺失 → 🟡
    - **手势交互规范**：拖拽阈值、惯性滚动参数、长按触发时间是否定义？缺失 → 🟡

**交互深挖层**（血肉——用户手指点下去系统怎么回应）：

*操作冲突与并发*：
- 同一操作重复触发时的行为（生成中再次点击同一阶段卡片 → 阻止/排队/覆盖？）
- 不同操作交叉触发时的行为（生成中点击其他阶段卡片 → 允许切换/阻止？）
- 中断操作后的状态回退（中断 AI 生成 → 阶段状态回退到 ⏹/保留 ⟳/进入新状态？）
- 失败重试的策略（API 失败重试 → 从头开始/断点续传？重试几次后放弃？）

*数据关联与完整性*：
- 文件重命名 → 关联的阶段绑定/文件映射是否自动更新
- 删除 Skill → 该 Skill 下已创建的项目如何处理（标记孤儿/禁止删除/提示迁移？）
- 导入 Skill 同名冲突 → 覆盖/重命名/跳过？哪种是默认、哪种需确认
- 恢复回收站文件 → 恢复路径、是否覆盖同名文件

*操作反馈与引导*：
- 长时间操作有进度指示（导入大 Skill、导出 Skill 打包、备份压缩）
- 首次用户不知道下一步时的引导（30 秒无操作 → 高亮提示可点击区域）
- 操作完成后的下一步引导（写完一章 → 是否需要引导写下一章？）
- 破坏性操作有二次确认（清空回收站、彻底删除、重新生成覆盖）

*辅助功能完整性*：
- 编辑器快捷键（Ctrl+S 保存、Ctrl+Z 撤销、Ctrl+Tab 切换标签、Ctrl+F 查找）
- 对话历史搜索/筛选能力

### D5 交互安全性

**结构层**（骨架——有没有定义策略）：
- 写者×文件矩阵：列出所有文件，标注所有可能的写者（用户编辑/AI生成/自动保存/外部修改），每格填冲突策略
- 入口×行为矩阵：列出所有打开文件的方式（阶段卡片/文件树/AI命令/快捷键），同一文件不同入口的行为是否一致或有约定
- 跨引擎桥接：Claude Engine ↔ Universal Engine 的 Session 格式兼容性 + 切换时的状态迁移方案
- 竞态条件：流式写入 + 用户编辑 + 自动保存三条路径的时序和优先级
- 数据持久化安全：API Key 加密存储、Session 自动保存、备份完整性、Trash 可恢复性

**交互深挖层**（血肉——操作副作用下的数据一致性）：

*关联断裂防护*：
- 文件重命名后，所有引用该文件的地方是否自动更新（阶段绑定、SKILL.md steps[].file、AI 上下文中的路径引用）
- Skill 被删除后，依赖该 Skill 的项目是否有降级方案（保留已有文件但失去 AI 方法论指导）
- 模板文件被删除后，Skill 的 `resources` 列表是否自动清理引用

*覆盖与冲突保护*：
- `[🔄 重新生成]` 覆盖文件前，是否检测到用户手动修改并提示"检测到手动修改，是否仍要覆盖？"（保护用户劳动）
- 导入 Skill 时，是否检测到同名 Skill 并给出"覆盖/保留两者/取消"三选一
- 外部磁盘操作（用户手动删了文件）被系统感知后，是否安全降级而非崩溃

*状态迁移安全*：
- AI 生成中途关闭应用 → 重启后文件是半成品还是空？阶段状态如何判定？
- 跨引擎切换时，Session 格式转换失败 → 是降级为"AI 失去上下文"还是保留从上一个引擎的摘要？
- 备份恢复时，恢复操作是否先验证备份文件完整性再覆盖当前数据

*UI 交互防呆*：
- 发送按钮在输入框为空时是否 disabled（防止发送空消息）
- 窗口关闭按钮是否在"有未保存内容"或"有 AI 生成进行中"时给出提示
- 面板拖拽是否有最小/最大宽度限制（防止用户把编辑区拖到 0px）
- **跨页面窗口行为一致性**：所有独立页面（非主工作台视图）是否包含窗口拖拽区域 + 最小化/最大化/关闭按钮。与主窗口行为不一致 → 🟠

## 最低通过标准

**结果判定**：审计通过 ≠ 打勾完成。审计通过的真正标准是——**任何一个拿到这份设计稿的开发者，能否不经追问、不看 Spec、不猜意图，直接写出像素级一致的代码。** 如果不能，说明设计稿还有信息缺口，审计不通过。

**量化标准**：
- 🔴 阻断：0 项（必须全部修复）
- 🟠 高风险：0 项（必须全部修复）
- 🟡 中风险：≤ 3 项，且每项标注处理策略（修复/延期/接受）
- 审计报告写入 Product-Spec-CHANGELOG.md，附时间戳和审计范围
- 未通过 → 修复 → 增量审计 → 重新判定，直到通过
- **Phase Gate 标记**：审计通过后逐层运行 `bash .claude/hooks/mark-gate-item.sh audit_L<N> pass`（L1-L5 各自独立标记，缺失层标 skip）。未经标记则 stop-gate.js 在 session 停止时阻止

**开发者自足性检查（最终关闸）**：
审计通过前，逐项确认以下问题答案均为"是"：
- 所有按钮的默认/hover/active/disabled 四态是否都有设计或明确的文字规范？
- 所有动态内容区域（列表/编辑区/对话区）是否有滚动条或溢出处理方案？
- 所有弹窗/对话框/右键菜单是否有完整设计，包括确认和取消两种路径？
- 所有页面跳转链路是否闭环（能去能回，不会迷路）？
- 所有字体/颜色/间距是否使用设计变量而非硬编码数值？
- 所有占位内容是否使用真实中文文本而非 Lorem ipsum？
