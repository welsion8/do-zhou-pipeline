[任务]
    读取 Product-Spec.md 和 Design-Brief.md，通过设计工具 MCP 生成完整的设计交付物。确保 Product Spec 中每个有 UI 的功能都有对应的设计页面，每个页面覆盖所有关键状态变体。

[文件结构]
    ```
    design-maker/
    └── SKILL.md                           # 主 Skill 定义（本文件）
    ```
    设计产出物（.pen 文件、导出图片）存放于项目根目录，不在此 Skill 目录内。

[依赖检测]
    Skill 启动时第一步自动执行。

    必需：
    - Product-Spec.md → 缺失则提示先调用 /product-spec-builder
    - Design-Brief.md → 缺失则提示先调用 /design-brief-builder
    - 设计工具 MCP → 见下方设计工具检测流程

    设计工具检测流程：
    1. 询问用户使用 Pencil 还是 Figma
    2. 检测对应 MCP 是否已连接
    3. 已连接 → 继续完整设计模式（.pen 文件）
    4. 未连接 → 尝试连接 MCP，或提示用户连接
    5. 用户未安装对应设计软件 → 进入**文本设计模式**
    6. 用户明确选择跳过 → 退出 design-maker，无设计稿

    **降级：文本设计模式**（MCP 不可用时）：
    - 从 Product-Spec.md + Design-Brief.md 提取页面清单、组件清单、状态变体清单
    - 生成 `Design-Spec.md`（纯文本设计规范），包含：
      - 所有页面的 ASCII wireframe + 布局描述
      - 每个 UI 区域的精确数值（宽度/高度/间距/字号，从 Design-Brief 和 Spec 推导）
      - 可复用组件的属性表（props、states、variants）
      - 颜色映射表（设计变量名 → hex 值）
    - 标注 `[无设计稿模式]`，后续 dev-builder 仅能依据文本描述编码
    - 此模式的审计 L3 层全部标记 `SKIP: 文本设计模式无 .pen 文件`

[第一性原则]
    **结果导向原则**：所有设计工作的最终评判标准只有一个——设计稿能否直接交付给开发团队，开发能否不经追问直接编码实现，最终产品能否对标市面上成熟的桌面应用（Linear / Notion / VS Code / Arc）的视觉品质。通过的标志不是你打完了所有勾，而是开发者拿到设计稿后不需要来问"这个按钮 hover 什么颜色""这个列表超出高度了怎么办""标签太多怎么排列"。所有答案都已在设计稿中。

    **完整覆盖原则**：Product Spec 中每个有 UI 的功能都必须有设计页面。漏一个页面，开发时就少一个参照，后果是开发靠猜。
    **状态完备原则**：每个页面不只有默认态。空状态、加载态、错误态、激活态，有交互的页面必须覆盖关键状态变体。
    **组件先行原则**：先建可复用组件，再用组件拼页面。避免同一个按钮在 10 个页面里画 10 遍、改一次要改 10 处。
    **文档驱动原则**：一切设计决策来自 Product-Spec.md 和 Design-Brief.md。不凭个人偏好发挥，不跳出文档范围添加文档未描述的功能。

[技能]
    - **文档分析**：从 Product Spec 提取所有页面、功能、交互元素；从 Design Brief 提取视觉方向
    - **设计规划**：将提取的信息转化为设计交付清单，列出所有需要设计的页面和变体
    - **组件设计**：使用设计工具 MCP 创建可复用组件系统
    - **页面设计**：使用设计工具 MCP 逐页面生成完整设计
    - **完整性校验**：对照 Product Spec 验证所有页面和状态是否覆盖

[设计交付物]
    一套完整的设计稿必须包含以下内容：

    **1. 设计变量**
    从 Design-Brief.md 提取并设置到设计工具中：
    - 颜色系统：背景色、文字色、品牌色、语义色、标签色
    - 字体系统：字体族、字号层级、字重
    - 间距系统：padding、gap 的常用值
    - 圆角系统：各层级的圆角值

    **2. 可复用组件**
    从 Product Spec 的 UI 布局和功能需求中提取通用组件：
    - 按钮（主要、次要、文字按钮）
    - 输入框
    - 导航项（选中态、未选中态）
    - 卡片
    - 标签/徽章
    - 其他页面中重复出现的元素

    **3. 所有页面**
    Product Spec UI 布局章节中描述的每个页面或视图都必须有对应设计：
    - 从 Product Spec 的 UI 布局、功能需求、用户流程三个章节交叉提取页面列表
    - 每个页面使用可复用组件拼装
    - 布局、间距、内容严格对照 Spec 描述

    **4. 状态变体**
    每个页面根据其交互复杂度覆盖对应状态：
    - 默认态：所有页面必须有
    - 空状态：有数据展示的页面必须有
    - 加载态：有异步操作的页面必须有
    - 错误态：有可能失败的操作必须有
    - 交互变体：同一区域有多种内容切换时，每种内容一个变体

[工作流程]
    [启动阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：加载文档
            读取 Product-Spec.md → 提取所有页面、功能、UI 布局描述、用户流程
            读取 Design-Brief.md → 提取情绪关键词、色彩方向、信息密度、排版方向、交互风格、核心页面视觉备注、状态设计方向

    [规划阶段]
        第一步：提取页面清单
            从 Product Spec 的 UI 布局章节提取所有描述的页面和视图
            从功能需求章节补充可能遗漏的页面
            从用户流程章节确认页面之间的跳转关系

        第二步：确定状态变体
            对每个页面分析需要哪些状态变体
            列出完整的设计交付清单，格式为：页面名 + 需要的状态变体列表

        第三步：提取组件清单
            从页面清单中识别重复出现的 UI 元素
            确定需要创建的可复用组件列表

        第四步：展示设计计划
            向用户展示完整的设计交付清单：
            - 组件数量
            - 页面数量
            - 变体数量
            - 总计设计项数
            用户确认后开始设计

    [设计阶段]
        第一步：获取设计工具指南
            调用设计工具的 get_guidelines 获取设计工具的使用规范和最佳实践

        第二步：设置设计变量
            根据 Design Brief 的色彩、字体、间距方向，通过设计工具 API 设置全局设计变量

        第三步：创建可复用组件
            按组件清单逐个创建
            每个组件创建后截图验证

        第四步：逐页面设计
            按页面清单逐个设计
            每个页面：
            1. 使用可复用组件拼装
            2. 填充真实内容，不使用 Lorem ipsum
            3. 对照 Product Spec 的描述逐项确认布局和内容
            4. 对照 Design Brief 的视觉备注确认风格
            5. 截图验证

        第五步：设计状态变体
            按变体清单逐个设计
            每个变体基于对应页面的默认态修改

    [校验阶段]
        设计产出后必须执行以下四阶段校验。对标业界设计 QA 标准。

        **第一步：完整性校验（有没有）**
        对照交付清单逐项确认：组件/页面/状态变体是否全部创建。Skill 编辑器所有标签页是否有对应设计。

        **第二步：布局合规性校验（对不对）**
        通过 snapshot_layout 和逐帧截图执行：
        - 全屏页面帧尺寸统一（如 1200×800），弹窗/内嵌视图尺寸合理
        - 顶层帧画布定位互不重叠，间距 ≥ 40px
        - 帧内子元素不溢出父帧（clip 帧除外），滚动区域有滚动条指示
        - **动态内容溢出防护**：识别所有承载用户输入或 AI 生成内容的区域（编辑区、对话区、列表区、标签栏、文件树、预览区、回收站），验证是否有滚动条或 overflow 处理
        - 绝对定位元素（窗口按钮）不与 flex 内容重叠
        - 输出：帧尺寸/位置清单 + 重叠/溢出问题列表 + 动态内容区域滚动条覆盖清单

        **第三步：视觉规范性校验（好不好）**
        逐帧截图对照 Design Brief 验证：
        - 同类型输入框/卡片/按钮宽度统一，左右边缘与参照元素对齐
        - 同类元素间距遵循 spacing 体系（sm/md/lg/xl/xxl）
        - 所有节点有语义化中文名（非"Frame 123"），组件带"组件-"前缀
        - 使用真实中文内容（非 Lorem ipsum），文案与 Spec 一致
        - 同一可复用组件在不同页面中外观一致（颜色/字号/圆角/间距）
        - 颜色来自设计变量（$text-*/$bg-*/$border-*/$accent-*），无硬编码色值

        **第四步：交互完备性校验（能不能用）**
        对照 Spec 和 Design Brief 逐条验证：
        - 可点击元素有视觉标识（颜色/hover 态/光标样式）
        - 空态/默认态/hover/激活/禁用/错误态有设计或文字说明
        - 开关/下拉/切换等组件两端状态齐全
        - 点击 A→跳转 B 的过渡有设计覆盖
        - 窗口控制按钮位置符合目标平台规范，工具栏含 drag region
        - 加载/错误/成功反馈有视觉设计

        **第五步：输出审计报告**
        汇总四阶段结果：✅ 通过项 / ⚠️ 警告项 / 🔴 失败项 + 修复建议。
        全部通过 → 引导 /dev-planner。有失败项 → 建议修复后再进入。
