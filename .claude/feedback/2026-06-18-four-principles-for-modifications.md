# 修改操作四原则

## 规则
任何对工程文件的修改、改动、修复，执行前必须过以下四条检查：

### 1. 链路完整性
任何改动必须倒查对上游（Spec → Design → Plan → Code）和下游的影响，不局部修补。
- 改代码前：这个改动是否与 Spec 一致？是否与 Design Brief 一致？是否在 DEV-PLAN 中有对应？
- 改 Spec 前：这个改动会影响哪些下游文件（Design Brief、DEV-PLAN、代码）？
- 不允许"只改这里不管其他地方"的局部修补。

### 2. 兼容性
改动不能破坏现有 workflow 文件、Skill 调用链、Agent 调度规则、审计矩阵的已有逻辑。
- 修改任何 .md 文件前：确认不会打断 Skill 触发条件、Agent 派发规则、审计执行流程。
- 修改 CLI 脚本/hook 前：确认不会破坏 Phase 关闸、审计管线、stop-gate 的运行。
- 改规则前先看谁在用——搜索所有引用点。

### 3. 工程文件冗余/互斥
新增规则不重复已有规则，不与现有规则冲突。优先修改源头（CLAUDE.md），不在 Skill 内重复定义。
- CLAUDE.md 是规则权威源——能放 CLAUDE.md 的规则不重复写在 Skill SKILL.md 中。
- 新增 Skill 前检查：这个规则是否已在 CLAUDE.md 或其他 Skill 中定义？
- Skill SKILL.md 引用全局规则而非重定义（已写入 CLAUDE.md 的"规则权威源"条款，本原则是对该条款的强化执行）。

### 4. 通用性
解决方案不绑定当前项目。换一个产品，同一套 agent 工作流仍能跑通。
- 不在规则中硬编码产品名、项目名、领域术语。
- 规则用"产品"而非"小说工具"，用"项目"而非具体项目名。
- 如果规则只对当前产品有效 → 归入 Product-Spec.md，不放入 agent 工作流文件。

## 触发场景
- 修改 CLAUDE.md、任何 SKILL.md、Agent 定义文件
- 修改 hook 脚本、审计管线
- 修改 workflow 文件
- 新增/修改 Skill 包
- 用户提出"改一下规则"、"加个限制"、"这里不对"等修正

## Why
用户观察到 agent 工作流在多次迭代中逐渐膨胀，出现规则重复定义、项目特定逻辑混入通用框架、局部修改破坏其他地方的问题。四原则是防止工程质量退化的事前门禁，成本远低于事后修复。

## How to apply
每次修改操作前，在思考中逐条过四原则：
1. 上游/下游影响范围是哪些文件？
2. 会不会打断已有的 Skill/Agent/审计/关闸逻辑？
3. 是不是在重复已有规则？能不能改 CLAUDE.md 而不是 Skill？
4. 换个产品（比如从"小说工具"换成"项目管理工具"）这条规则还成立吗？
