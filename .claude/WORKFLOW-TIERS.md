# 工作流分级

> 同一套 agent 框架，两种深度。按项目预算和产品阶段选择。

---

## 标准版（Standard）

**适用场景**：MVP、内部工具、个人项目、早期验证、预算敏感
**Spec 输出**：~300 行 Product-Spec.md
**设计交付物**：设计变量 + 组件 API + 深色模式设计帧
**审计深度**：5×5 矩阵 25 单元 + 交互深挖 28 项

**包含能力**：
- 功能需求结构化 + 状态机 + 入口×行为矩阵
- 并发冲突策略 + 滚动防冲突 + 双引擎桥接
- 设计变量系统 + 组件 API 文档（变体/状态/代码示例）
- 4 断点响应式适配 + 空态/加载态/错误态
- 5×5 交叉审计 + 开发者自足性关闸
- CI/CD + 测试覆盖率 + 签名配置 + 错误监控模板（自动生成）
- 主题可访问性检查（深色 + 浅色反转策略）

**Token 预算参考**：Spec 阶段 ~15K-25K tokens，设计阶段 ~20K-35K tokens

---

## 专业版（Professional）

**适用场景**：面向终端消费者的付费产品、需要量投的桌面/Web 应用、对标 Linear/Obsidian/Notion 体验水准
**Spec 输出**：~500+ 行 Product-Spec.md
**设计交付物**：标准版全部 + 动效语义表 + 可访问性内建验证
**审计深度**：标准版全部 + 专业版专项（性能预算/动效降级/对比度校验）

**在标准版基础上新增**：

### Spec 增强
- **反功能需求**：「不做」列表——明确声明哪些需求被评估后放弃及理由
- **性能预算表**：核心操作 P50/P90/P99 目标 + CI 红线（任一指标 P90 超预算 → 构建失败）
- **降级路径时序图**：关键并发路径（用户编辑→自动保存→AI 写入）在时间轴上的交错点 + 每点的行为决策
- **恢复策略详解**：API 失败退避参数（间隔/次数/每次的 UI 态变化）

### 设计系统增强
- **动效语义表**：按用途分类（入场/离场/注意/状态过渡）+ 编排规则（stagger 间隔、父→子延迟、离场优先）
- **prefers-reduced-motion 降级**：所有动效的即时切换替代方案
- **可访问性内建**：设计 tokens 自带对比度校验（WCAG AA/AAA）+ focus-ring 样式 + `$text-tertiary` 等边界色自动标记
- **手势交互规范**：拖拽阈值、惯性滚动参数、长按触发时间

### 审计增强
- 性能预算 CI 验证
- 对比度自动计算（token 生成时即校验，非审计时补救）
- reduced-motion 覆盖检查

**Token 预算参考**：Spec 阶段 ~30K-50K tokens，设计阶段 ~40K-70K tokens

## 选择指南

| 项目类型 | 推荐版本 |
|---------|---------|
| 自己用的工具 / 内部工具 | 标准版 |
| 给朋友用的小产品 | 标准版 |
| 面向消费者的付费产品 | 专业版 |
| 有投资人/需要展示给外部的产品 | 专业版 |
| 追求极致体验的对标产品 | 专业版 |
| 快速验证想法 | 标准版（验证后再升级 Spec） |

---

## 版本切换

在 CLAUDE.md 中设置：
- `工作流级别: 标准版` → 按标准版执行
- `工作流级别: 专业版` → 按专业版执行（完整 Spec + 动效 + a11y 内建）

默认级别：**专业版**（因为你在做面向消费者的写作工具）

---

## Phase 验证差异

| 验证项 | 标准版 | 专业版 |
|--------|--------|--------|
| 编译证 (tsc + build) | ✅ 必须 | ✅ 必须 |
| 功能证 (交付清单打勾 + 路径验证) | 自查即可 | ✅ 必须，附证据 |
| 审查证 (code-reviewer sub-agent) | 不需要 | ✅ 交互密集 Phase 必须 |
| 视觉证 (设计稿↔代码逐组件对照) | 不需要 | ✅ UI Phase 必须 |
| Phase 关闸硬度 | 缺证不放行 | 缺证不放行 |
